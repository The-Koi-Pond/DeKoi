Architecture

Move toward the old De-Koi skeleton: app, layered features, shared APIs, engine, host.

DeKoi uses the old De-Koi architecture as the target because it is stronger than a flat feature tree: ownership stays visible, runtime and database details stay out of UI, and product rules stay portable.

The current repo is still migrating. Treat today's flat feature folders, src/runtime, and src/features/navigation as bridge layers, not the final shape.

Target Source Shape

Path Owner Role
src/app App composition React bootstrap, providers, startup effects, and app-level wiring.
src/features/catalog Resource/data owners Companions, personas, lorebooks, provider connections, thread libraries, presets, galleries, and importable collections.
src/features/runtime Shared frontend runtime systems Generation flow, ripples, visuals, trackers, and mode-neutral live state used by shell and modes.
src/features/modes User mode surfaces Messenger, Classic, future adventure/gameplay modes, shared mode UI, and the mode router.
src/features/shell App-level tools Pond shell, care/settings, data backup, provider setup, navigation chrome, and desktop integration UI.
src/shared Generic frontend helpers Browser utilities, primitives, and UI components with no concrete feature ownership.
src/shared/api Typed runtime wrappers Desktop Tauri command and remote runtime HTTP wrappers. Features may call these; engine code may not.
src/engine Product engine React-free records, pure actions, selectors, generation contracts, and capability ports.
src-tauri Native host Local storage, secrets, provider transport, command dispatch, and remote runtime host behavior.

Current Bridge Map

Current path Bridge role Target direction
src/App.tsx, src/main.tsx Provider wiring and first render composition. Move app composition into src/app.
src/features/navigation App state, persistence sync, action hooks, import/export, and provider/controller logic. Move provider/startup ownership toward src/app and mode routing ownership toward features/modes.
src/runtime Storage contracts, desktop/remote runtime transport, import/export normalization, app settings, and generation adapters. Split typed transport wrappers into src/shared/api and shared frontend systems into features/runtime.
src/features/classic, src/features/messenger, src/features/pond Current flat feature screens. Move mode surfaces under features/modes and shell surfaces under features/shell.
src/features/catalog, src/features/shell Folders that already resemble the target layers. Deepen into owner packages and public entrypoints as they grow.

Enforced Import Boundaries

Run the boundary check when imports or package structure change:

pnpm check:frontend-boundaries
Boundary Reason
src/engine must not import runtime, features, shared frontend helpers, React, or Tauri packages. The engine must stay portable, testable, and free of storage or UI concerns.
src/runtime must not import src/features or React. The bridge runtime exposes capability contracts; it does not call screens or hooks.
src/shared must not import features; generic shared code outside src/shared/api must not import engine or runtime adapters. Shared helpers stay generic, while shared/api remains the typed runtime wrapper exception.
Old-shape feature layer direction is shell -> modes -> runtime -> catalog. Higher layers compose lower layers; lower resource/runtime layers do not depend on app screens.
src/features/navigation must not import sibling feature UI modules. The navigation bridge owns orchestration and state, not concrete screen composition.
Non-navigation features must not add new direct src/runtime imports. Existing runtime callers are frozen as legacy exceptions while they move behind app/runtime APIs.

Migration Direction

  1. Create src/app and move provider/startup composition out of root files.
  2. Move navigation provider/controller ownership toward app providers and mode router boundaries.
  3. Move Messenger and Classic screens under features/modes.
  4. Move Pond shell, care drawer, shoal, tide, bank, and waterline under features/shell.
  5. Split current src/runtime into typed src/shared/api wrappers and features/runtime systems where appropriate.
  6. Deepen features/catalog into resource-owned packages as collections grow.
  7. Add stricter private-folder and public-entrypoint checks once those packages exist.

Storage And DB Direction

Storage owns persistence mechanics. Engine owns product meaning.

  • Durable record names and shapes stay DeKoi-native.
  • Runtime/shared API modules normalize JSON, call desktop/remote storage, and hide future SQLite or database implementation details.
  • Engine modules define records and mutations without knowing how records are stored.
  • App/runtime orchestration loads, syncs, imports, exports, and exposes typed actions to the UI.
  • Feature UI consumes typed actions or narrowly scoped props; it should not learn DB tables, host commands, or storage file layout.
  • Future DB adapters implement runtime repository/storage functions behind existing contracts.

Related Checks

Command Purpose
pnpm check:storage-contracts Keeps TypeScript storage entities and the Rust desktop allowlist aligned.
pnpm check:runtime-contracts Keeps TypeScript runtime commands, Rust runtime dispatch, and the remote fixture aligned.
pnpm check:desktop-contracts Keeps desktop command names aligned.
pnpm check:frontend-boundaries Keeps frontend import direction aligned.

If a convenient import crosses ownership, add or adjust the owning layer contract instead of reaching across the boundary.

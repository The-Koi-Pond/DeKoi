Agent onboarding

Ask a coding agent “how do I get started?” and use this path.

This page gives coding agents and developers the same starting point: run the docs, learn the repo shape, run the current Vite app, test real workflows, then fix bugs from the correct owner.

Current status: Messenger, Roleplay, catalog records, collection storage, native bundles, provider secrets, and the remote runtime fixture all exist in early form. Media, presets, Reserved, and a routed Ripple editor are still placeholders or incomplete.

First Agent Response

When asked how to get started, a coding agent should do this:

  1. Start the docs server with pnpm docs:dev.
  2. Give the user http://127.0.0.1:4174/.
  3. Teach the repo shape in a short layered tour.
  4. Show how to run the current Vite app.
  5. Guide a manual testing pass through Messenger, Roleplay, catalog, and storage paths.
  6. When a bug is found, switch into root-cause bug fixing.

Manual Testing Path

Messenger

Create a Companion, Persona, and Connection. Start a Messenger thread, send a message, generate a reply, edit/delete a message, reload, and continue.

Roleplay

Create a Roleplay thread, send an entry, generate a character turn, edit/delete an entry, reload, and continue.

Catalog

Create, edit, duplicate, and delete Companions, Personas, Lorebooks, and Connections. Watch referenced threads for cleanup or missing states.

Placeholders

Confirm Reserved is locked, Media and Presets rails are placeholder-only, and Ripples have no routed editor yet.

Settings and imports

Create provider connections, test them, import/export DeKoi bundles, verify pre-import backups, and check visible provider errors for API key, Base URL, selected model, network, and unsupported-provider cases.

Remote runtime

Run pnpm runtime:fixture, configure its URL in Pond Care > Deep Water, and test collection storage and runtime command health through the invoke contract.

Coding agents should not start coding from a vague onboarding request. They should run the docs, teach the repo, help the user test the app, then edit code only after there is a concrete bug or feature request.

Use AGENTS.md, CORE_FEATURES_HANDOFF.md, and these developer docs together when orienting in the repo.
